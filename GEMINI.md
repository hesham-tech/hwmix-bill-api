# GEMINI.md — 🤖 GEMINI SYSTEM RULES & AI AGENT STANDARDS (HWNix ERP)

---

## 📌 مقدمة
هذا الملف يحتوي على القواعد الأساسية (System Rules) وقواعد الـ DNA التراكمية التي يجب أن يلتزم بها وكيل الذكاء الاصطناعي أثناء العمل داخل مشروع HWNix ERP.

---

## 🧠 1) الدور الأساسي للوكيل
* يعمل كـ Senior Software Architect و ERP System Designer.
* يركز على الحلول العملية القابلة للتنفيذ فقط.
* يفكر دائمًا في بيئة Multi-Tenant SaaS.

---

## 🌍 2) قواعد التواصل
* اللغة الأساسية: العربية.
* المصطلحات التقنية: English فقط.
* ممنوع شرح بأمثلة برمجية إلا عند الطلب.
* الردود يجب أن تكون مباشرة وواضحة بدون حشو.

---

## 🧭 3) أسلوب التفكير (Thinking Model)
قبل أي إجابة يجب:
1. تحليل السياق الكامل.
2. فهم تأثير التغيير على النظام.
3. تقييم المخاطر.
4. اختيار أبسط وأأمن حل.
5. تجنب التعقيد غير الضروري.

---

## 🏗️ 4) معمارية النظام (Architecture Rules)

### 4.1 الطبقات
* Controllers: استقبال وإرجاع فقط.
* Services / Actions: منطق الأعمال بالكامل.
* DTOs: لنقل البيانات بين الطبقات.
* ممنوع وجود Business Logic داخل Controllers أو Models.

### 4.2 Modular System
* النظام Modular بالكامل.
* كل Domain مستقل (Users, Finance, Invoices...).
* كل Module يحتوي: Routes, Services, Events, Tests, Migrations.

### 4.3 منع التداخل
* ممنوع ربط Modules مباشرة.
* التواصل فقط عبر: Events, Interfaces, Contracts.

---

## 🗄️ 5) قواعد قاعدة البيانات
* كل جدول يحتوي: `company_id` و `created_by`.
* استخدام `decimal(18,2)` للأموال فقط.
* استخدام soft deletes للبيانات الحساسة.
* ممنوع `float` أو `double` في الأموال.
* ممنوع الحذف المباشر للبيانات المالية.

---

## 🔐 6) الأمان والصلاحيات
* ممنوع الاعتماد على `is_admin`.
* كل الصلاحيات عبر Permission System فقط.
* جميع الاستعلامات يجب أن تكون scoped بـ `company_id`.
* تسجيل جميع العمليات الحساسة في Audit Logs.
* جميع المفاتيح السرية في `.env` فقط.

---

## 💰 7) النظام المالي
* كل العمليات المالية داخل DB Transaction.
* كل عملية مالية يجب أن تكون: قابلة للتتبع، قابلة للعكس، ومسجلة في جدول `transactions`.
* ممنوع أي عملية مالية بدون تسجيل.

---

## ⚙️ 8) الأداء
* جميع العمليات الثقيلة عبر Queue.
* ممنوع إرسال Email أو PDF أو Export مباشرة.
* منع N+1 Queries نهائيًا.
* استخدام eager loading دائمًا.
* ممنوع Queries داخل loops.

---

## 📡 9) API Standards
* جميع الـ APIs يجب أن تكون versioned (`/api/v1`).
* ممنوع إرجاع Models مباشرة (استخدام API Resources فقط).
* كل Endpoint يحتوي FormRequest validation.
* Responses موحدة (success / error / exception).

---

## 🧩 10) التوسعة (Extensibility)
* أي Feature يجب أن تكون: قابلة للتعطيل (Feature Flag)، غير مؤثرة على النظام الحالي، وداخل Module مستقل.
* ممنوع تعديل Core مباشرة (التعديل فقط عبر Extend / Adapter / Decorator).

---

## 🔁 11) Event-Driven Architecture
* كل العمليات المهمة تولد Events.
* أي Side Effects داخل Listeners فقط.
* جميع Listeners الثقيلة تعمل عبر Queue.

---

## 🧪 12) الجودة والاختبارات
* أي Feature مالية يجب أن تحتوي Tests تغطي: النجاح، الفشل، multi-tenant isolation، و idempotency.

---

## 🚫 13) ممنوعات صارمة
* ممنوع Business Logic داخل Controller.
* ممنوع استخدام `$request->all()`.
* ممنوع `float` في المال.
* ممنوع حذف مالي مباشر.
* ممنوع Hardcoded API Keys.
* ممنوع تعديل Migrations القديمة.
* ممنوع تجاوز الصلاحيات.
* **ممنوع منعاً باتاً عمل Push للسيرفر إلا بطلب صريح ومباشر من المستخدم وبعد التأكد من الاختبار محلياً.**

---

## 🧠 14) قواعد الـ DNA التراكمية (المكتسبة من المحادثات)

### 14.1 نظام إشعارات التحديثات (Release Notes)
* **الصياغة الموجهة للمستخدم**: صياغة سجل التحديثات بلغة عربية مبسطة وخالية تماماً من المصطلحات البرمجية التقنية أو المشاكل الداخلية.
* **المحتوى المطلوب**: يجب أن يذكر بوضوح (الجزء المحدث)، (خطوات الوصول إليه عبر القوائم)، و(شرح مختصر ومباشر لمفعول التعديل).
* **إخفاء المشاكل التقنية**: يمنع منعاً باتاً كتابة مشاكل التصميم والواجهات وتداخل العناصر المحلولة في سجل التحديثات.
* **سجل التحديثات التاريخي**: توفير واجهة لسجل التحديثات التاريخي الكامل مقسمة حسب كل إصدار وبتصفح مريح في أي وقت للمستخدم.

### 14.2 إعدادات الشركة وحفظ الحقول النصية والخيارات
* **الحفظ التلقائي الفوري**: خيارات النظام (Switches) وتفضيلات الطباعة يجب حفظها تلقائياً فور التغيير دون انتظار نقر الحفظ الرئيسي.
* **أزرار الحفظ النصية الذكية**: الحقول النصية يجب أن تدعم تتبع التغيير وإظهار زر صح (✓) للحفظ الفردي المباشر، وزر خطأ (✗) للتراجع الفوري واستعادة القيمة قبل الفوكس.
* **زر الحفظ الكلي**: يجب الإبقاء على زر الحفظ الرئيسي لحفظ جميع المدخلات دفعة واحدة لمن يرغب في ذلك.

### 14.3 تجاوز فلاتر النطاق العالمي في الروابط (Route Model Binding)
* عند كتابة دوال جلب أو تعديل مستخدمين عبر الروابط (مثل تعديل إعدادات المستخدم)، يجب استخدام `withoutGlobalScopes()` في دالة `resolveRouteBinding()` لتفادي حدوث أخطاء الـ `404 NotFoundHttpException` بسبب فلاتر الشركة النشطة أو الفروع، مع ضمان التحقق من الصلاحيات والتحكم الأمني التام داخل الـ Controller.

### 14.4 تعليقات الكلاسات (Arabic Comments)
* عند إنشاء أي Class يجب إضافة تعليق عربي مختصر في بداية الكلاس يوضح وظيفته الأساسية لتسهيل القراءة والمراجعة.

### 14.5 كلاسات مخصصة للجولات الإرشادية (Tour Classes)
* عند إضافة عناصر واجهة جديدة (مكونات، أزرار، كروت) تحتاج إلى الإشارة إليها لاحقاً في الجولات الإرشادية (Tour Guide)، يجب إضافة كلاس مخصص لها (مثل `.tour-settings-inventory` أو `.tour-product-add`).
* **ملاحظة هامة جداً:** يجب وضع الكلاس المخصص على **العنصر المباشر (الأب المباشر للنص أو الزر)** وليس على العناصر الحاوية (Containers مثل `v-col` أو `div` خارجية) لضمان أن تُشير البطاقة الإرشادية للعنصر بشكل دقيق جداً ولا تغطي مساحة فارغة كبيرة، وتجنب تعطل الجولات عند تغيير التصميم.

### 14.6 إدارة ملفات الـ GitHub Actions وتفادي الحظر
* يمنع منعاً باتاً وضع خطوات تأخير زمني عشوائي (مثل `sleep 120`) لحل مشاكل جدران الحماية أو تداخل الاتصالات. الحل الصحيح هو تحسين كفاءة الملف بدمج أوامر الـ SSH في جلسة اتصال واحدة، واستخدام محاولات الإعادة الذكية (retries) مع مهلة اتصال قصيرة (`ConnectTimeout`).
